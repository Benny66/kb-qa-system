## 第一章 绪论（约 1500 字）

### 1.1 研究背景与意义

**写作思路**：从大背景切入，说明为什么要做这个系统。

**可用素材（直接改写）**：

> 随着互联网和数字化技术的快速发展，各类组织积累了海量的文档资料，如何高效地从这些文档中检索和获取所需信息，成为信息管理领域的重要课题。传统的关键词检索方式存在语义理解能力弱、无法处理自然语言提问等局限性。
>
> 近年来，以 GPT、GLM 为代表的大语言模型（Large Language Model，LLM）取得了突破性进展，其强大的自然语言理解和生成能力为智能问答系统的构建提供了新的技术路径。将大语言模型与私有知识库相结合，构建能够基于特定文档内容进行精准问答的系统，具有重要的理论价值和广泛的应用前景。
>
> 本文设计并实现了一个基于大语言模型的知识库智能问答系统，用户可以上传自定义的文本文档作为知识库，系统通过 Prompt 工程将知识库内容注入大模型上下文，实现基于文档内容的精准问答，有效解决了大模型"幻觉"问题，提升了问答的准确性和可信度。

### 1.2 国内外研究现状

**写作思路**：简要介绍同类系统，说明本系统的差异化。

**可用素材**：

> 在智能问答领域，国外已有 ChatPDF、Notion AI 等成熟产品，国内也出现了基于文心一言、通义千问的企业知识库产品。这些系统大多面向企业级用户，部署复杂、成本较高。
>
> 本系统面向轻量化应用场景，采用开源框架构建，支持本地部署，具有成本低、易扩展、隐私安全等优势，适合中小型团队或个人用户使用。

## 第二章 相关技术介绍（约 2000 字）

### 2.1 大语言模型与 Prompt 工程

**写作要点**：
- 什么是大语言模型（LLM）：基于 Transformer 架构，通过海量文本预训练
- 什么是 Prompt 工程：通过设计输入提示词引导模型输出
- 本系统如何使用：将知识库全文注入 System Prompt，约束模型只基于文档内容回答

**关键技术点（可展开写）**：
```
System Prompt 设计策略：
1. 角色设定：定义模型为"知识库问答助手"
2. 内容注入：将知识库文本嵌入 Prompt
3. 规则约束：明确要求模型不编造信息
4. 兜底处理：知识库无相关内容时告知用户
```

### 2.2 Vue 3 前端框架

**写作要点**：
- Vue 3 的 Composition API 相比 Vue 2 Options API 的优势
- Pinia 状态管理：集中管理用户登录态、全局消息提示
- Vue Router：前端路由管理，实现 SPA 单页应用，路由守卫保护需登录页面
- Vite：新一代前端构建工具，开发时热更新速度快

### 2.3 Flask 后端框架

**写作要点**：
- Flask 是 Python 轻量级 Web 框架，适合快速构建 RESTful API
- Flask-SQLAlchemy：ORM 框架，用 Python 类定义数据库表结构，无需手写 SQL
- Flask-JWT-Extended：JWT（JSON Web Token）无状态鉴权机制
- Flask-CORS：处理前后端分离场景下的跨域请求问题

### 2.4 RESTful API 设计规范

**写作要点**：
- REST 架构风格的核心原则：资源、统一接口、无状态
- 本系统接口设计遵循 RESTful 规范：GET 查询、POST 创建、DELETE 删除
- 统一响应格式设计：`{ code, msg, data }` 结构

### 2.5 SQLite 数据库

**写作要点**：
- SQLite 是嵌入式关系型数据库，无需独立服务进程
- 适合轻量级应用，数据存储为单个文件，便于部署和迁移
- 本系统使用 SQLAlchemy ORM 操作，后期可无缝切换至 MySQL/PostgreSQL

### 4.1 系统总体架构设计

**写作要点**：描述前后端分离架构。

**架构描述模板**：

> 本系统采用前后端分离的 B/S 架构。前端基于 Vue 3 框架构建单页应用（SPA），通过 Axios 发送 HTTP 请求与后端通信；后端基于 Flask 框架提供 RESTful API 服务，负责业务逻辑处理、数据库操作和 AI 服务调用；数据库采用 SQLite 存储用户数据、知识库元信息和问答历史；AI 服务通过调用智谱 AI 的 GLM 大模型接口实现智能问答功能。

### 4.5 AI 问答模块设计

**写作要点**：这是本系统的核心创新点，要重点描述。

**Prompt 设计策略**：

> 本系统 AI 问答模块的核心在于 Prompt 工程设计。系统将知识库 TXT 文件的全文内容嵌入 System Prompt，通过明确的角色设定和规则约束，引导大模型严格基于知识库内容进行回答，避免模型产生"幻觉"（即编造不存在的信息）。
>
> System Prompt 设计包含四个关键要素：
> 1. **角色设定**：将模型定义为"专业的知识库问答助手"
> 2. **内容注入**：将知识库全文以结构化方式嵌入 Prompt
> 3. **规则约束**：明确要求模型只基于知识库内容作答
> 4. **兜底策略**：当知识库无相关内容时，要求模型明确告知用户

**处理流程**：
```
用户提问
   ↓
读取知识库 TXT 文件（自动识别 UTF-8/GBK 编码）
   ↓
超长内容截断（超过 12000 字符自动截断）
   ↓
构建 System Prompt（注入知识库内容 + 规则约束）
   ↓
调用智谱 AI GLM 接口
   ↓
返回回答 + Token 消耗量
   ↓
保存至数据库历史记录
```

### 5.1 用户认证模块实现

**写作要点**：描述登录流程和 JWT 鉴权机制。

**关键代码说明**（引用 `app.py` 中的登录接口）：

> 用户登录时，后端使用 Werkzeug 库的 `check_password_hash` 函数验证密码哈希，验证通过后调用 `create_access_token` 生成 JWT Token 并返回给前端。前端将 Token 存储于 `localStorage`，后续每次请求通过 Axios 拦截器自动在请求头中携带 `Authorization: Bearer <token>`，后端通过 `@jwt_required()` 装饰器验证 Token 有效性。

### 7.2 不足与展望

**写作模板**：

> 由于时间和技术能力的限制，本系统还存在以下不足，有待后续改进：
>
> 1. **知识库格式单一**：目前仅支持 TXT 格式，未来可扩展支持 PDF、Word、Markdown 等格式
> 2. **检索方式简单**：当前采用全文注入方式，对于超长文档效果有限，未来可引入向量数据库（如 Chroma、Faiss）实现语义检索，提升大文档的问答精度
> 3. **用户管理简单**：目前仅支持预置账号，未来可增加用户注册、权限管理等功能
> 4. **无多轮对话记忆**：当前每次问答独立，未来可引入对话历史上下文，支持多轮连续对话
> 5. **部署方式单一**：目前仅支持本地部署，未来可容器化（Docker）并支持云端部署
