## 一、系统整体类

### ⭐ Q1：请简单介绍一下你的毕业设计

**参考回答**：

> 我的毕业设计是一个**基于大语言模型的知识库智能问答系统**。
>
> 核心功能是：用户可以上传自己的 TXT 文档作为知识库，然后用自然语言向系统提问，系统会调用智谱 AI 的 GLM 大模型，基于知识库的内容给出精准的回答，并保存完整的问答历史。
>
> 技术上采用前后端分离架构，前端用 Vue3，后端用 Python Flask，数据库用 SQLite，AI 部分接入了智谱 AI 的接口。整个系统可以本地部署运行。

---

### ⭐ Q2：你为什么选择这个题目？

**参考回答**：

> 主要有两个原因。
>
> 第一，大语言模型是当前 AI 领域最热门的方向，我想通过毕设实际动手接触这个技术。
>
> 第二，我发现大模型虽然知识丰富，但对于特定领域的私有文档，它并不了解，而且容易"编造"不存在的信息。我想解决这个问题——让大模型只基于用户自己的文档来回答，这样更准确、更可信。这个需求在实际工作中也很常见，比如企业内部文档查询、学习资料问答等场景。

---

### Q3：这个系统的应用场景是什么？

**参考回答**：

> 应用场景比较广泛。比如：
> - **学生学习**：上传课本或笔记，向系统提问，辅助复习
> - **企业内部**：上传公司规章制度、产品手册，员工可以快速查询
> - **个人知识管理**：上传自己整理的资料，随时问答检索
>
> 总体来说，凡是需要"基于特定文档快速找到答案"的场景，都可以用这个系统。

---

### Q4：系统有哪些不足？如果继续做，你会怎么改进？

**参考回答**：

> 主要有三点不足：
>
> 第一，**知识库格式单一**，目前只支持 TXT，实际使用中 PDF 和 Word 更常见，后续可以加入文档解析库来支持更多格式。
>
> 第二，**大文档处理能力有限**，当前是把整个文档塞进 Prompt，文档太长会被截断。更好的方案是引入向量数据库，先把文档切片、向量化存储，提问时只检索最相关的片段，这样可以处理更大的文档。
>
> 第三，**没有多轮对话记忆**，每次提问都是独立的，不能基于上一轮的回答继续追问。后续可以把对话历史也传给模型，实现连续对话。

## 二、技术选型类

### ⭐ Q5：你为什么选择 Vue3 而不是 React 或其他框架？

**参考回答**：

> 主要考虑了三点：
>
> 第一，Vue3 的学习曲线相对平缓，文档是中文的，上手更快。
>
> 第二，Vue3 的 Composition API 让代码组织更灵活，可以把相关逻辑放在一起，比 Vue2 的 Options API 更清晰。
>
> 第三，Vue3 配合 Vite 构建工具，开发时热更新速度非常快，开发体验好。
>
> React 也是很好的选择，但对于这个项目的规模，Vue3 已经完全够用，而且我对 Vue 更熟悉。

---

### ⭐ Q6：你为什么选择 Flask 而不是 Django？

**参考回答**：

> Flask 是轻量级框架，Django 是全栈框架。
>
> 我的项目是前后端分离的，后端只需要提供 RESTful API，不需要 Django 自带的模板引擎、Admin 后台这些功能。Flask 更轻量，配置更灵活，代码量更少，非常适合这种纯 API 服务的场景。
>
> 而且 Flask 的扩展生态很丰富，我用到的 JWT 鉴权、ORM、跨域处理都有对应的 Flask 扩展，按需引入就好。

---

### Q7：为什么用 SQLite 而不是 MySQL？

**参考回答**：

> SQLite 是嵌入式数据库，不需要单独安装数据库服务，数据存在一个文件里，部署非常简单，非常适合这种单机部署的轻量级应用。
>
> 对于这个系统的数据量，SQLite 完全够用。而且我用的是 SQLAlchemy ORM，如果以后需要换成 MySQL 或 PostgreSQL，只需要改一行数据库连接字符串，代码不需要改动，扩展性是有保障的。

---

### ⭐ Q8：JWT 是什么？你的系统是怎么用的？

**参考回答**：

> JWT 是 JSON Web Token，是一种无状态的身份认证机制。
>
> 传统的 Session 认证需要服务器存储用户状态，而 JWT 把用户信息加密编码在 Token 里，服务器不需要存储，只需要验证 Token 的签名是否合法。
>
> 我的系统里，用户登录成功后，后端生成一个 JWT Token 返回给前端，前端存在 localStorage 里。之后每次请求，前端通过 Axios 拦截器自动在请求头里加上 `Authorization: Bearer <token>`，后端用 `@jwt_required()` 装饰器验证 Token，验证通过才处理请求。
>
> 这样做的好处是前后端完全解耦，后端不需要维护 Session，也更容易水平扩展。

---

### Q9：什么是 RESTful API？你的接口是怎么设计的？

**参考回答**：

> REST 是一种接口设计风格，核心思想是把系统的数据和功能抽象为"资源"，用 HTTP 方法来表达对资源的操作。
>
> 具体来说：GET 表示查询，POST 表示创建，DELETE 表示删除，PUT/PATCH 表示修改。
>
> 我的系统里，比如知识库这个资源：
> - `GET /api/kb` 获取知识库列表
> - `POST /api/kb/upload` 上传新知识库
> - `DELETE /api/kb/1` 删除 id 为 1 的知识库
>
> 所有接口都返回统一的 JSON 格式：`{ code, msg, data }`，前端可以统一处理。

## 三、核心功能类

### ⭐ Q10：你的系统是怎么实现"基于知识库回答"的？

**参考回答**：

> 这是系统的核心，用的是 **Prompt 工程**的方法。
>
> 大语言模型的接口支持传入一个"系统提示词"（System Prompt），这个提示词会在对话开始前告诉模型它的角色和规则。
>
> 我的做法是：用户提问时，先读取知识库 TXT 文件的全部内容，然后把这些内容嵌入到 System Prompt 里，同时在 Prompt 里明确告诉模型："只能根据以下知识库内容回答，不能编造信息，如果知识库里没有相关内容，要明确告知用户"。
>
> 这样模型就被约束在知识库的范围内回答，有效避免了大模型"幻觉"的问题。

---

### Q11：什么是大模型的"幻觉"问题？你是怎么解决的？

**参考回答**：

> 大模型的"幻觉"是指模型在不知道答案的情况下，会自信地编造一个听起来合理但实际上错误的答案，这在需要准确信息的场景下是很危险的。
>
> 我的解决方案是通过 Prompt 约束：把知识库内容注入 System Prompt，并明确要求模型"只基于提供的内容回答，知识库中没有的信息不要回答"。同时设置了较低的 temperature 参数（0.3），让模型的输出更保守、更确定，减少随机发挥。
>
> 当然这个方案不是完美的，更彻底的解决方案是引入向量数据库做语义检索，但对于这个项目的规模，当前方案已经能满足需求。

---

### Q12：文件上传是怎么实现的？有没有做安全处理？

**参考回答**：

> 文件上传用的是 HTML 的 `<input type="file">` 配合 Axios 发送 `multipart/form-data` 请求。
>
> 安全处理做了三点：
> 1. **文件类型校验**：后端检查文件扩展名，只允许 `.txt` 格式，拒绝其他类型
> 2. **文件名安全处理**：用 Werkzeug 的 `secure_filename` 函数过滤掉文件名中的特殊字符，防止路径注入攻击
> 3. **UUID 重命名**：存储时在文件名前加 UUID 前缀，防止同名文件覆盖，也防止用户猜测文件路径

---

### Q13：密码是怎么存储的？

**参考回答**：

> 密码不是明文存储的，用的是 Werkzeug 库的 `generate_password_hash` 函数，它会对密码进行加盐哈希处理，存储的是哈希值。
>
> 验证时用 `check_password_hash` 函数，把用户输入的密码和数据库里的哈希值比对，即使数据库泄露，攻击者也无法直接得到用户的原始密码。

## 四、代码实现类

### ⭐ Q14：前端是怎么处理登录状态的？

**参考回答**：

> 分三个层面处理：
>
> 第一，**存储层**：用 Pinia 做状态管理，用户登录后把 Token 和用户信息存在 Pinia store 里，同时持久化到 `localStorage`，这样刷新页面不会丢失登录状态。
>
> 第二，**请求层**：用 Axios 拦截器，每次发请求前自动从 localStorage 读取 Token，加到请求头的 `Authorization` 字段里，不需要每个接口单独处理。
>
> 第三，**路由层**：用 Vue Router 的 `beforeEach` 路由守卫，每次跳转页面前检查是否已登录，没登录就跳转到登录页，防止未授权访问。

---

### Q15：前后端是怎么解决跨域问题的？

**参考回答**：

> 跨域问题是因为浏览器的同源策略，前端（localhost:5173）和后端（localhost:5001）端口不同，属于跨域请求。
>
> 我用了两种方式处理：
>
> 开发环境：在 Vite 的配置文件里设置了 `proxy` 代理，把前端发出的 `/api` 请求转发到后端，这样从浏览器角度看请求是同源的，不存在跨域。
>
> 生产环境：后端用 Flask-CORS 扩展，在响应头里加上 `Access-Control-Allow-Origin`，允许跨域请求。

---

### Q16：Pinia 和 Vuex 有什么区别？你为什么用 Pinia？

**参考回答**：

> Pinia 是 Vue3 官方推荐的状态管理库，是 Vuex 的继任者。
>
> 主要区别：Pinia 的 API 更简洁，不需要 mutations，直接修改 state；支持 TypeScript 更好；没有 Vuex 那么多模板代码；而且 Pinia 的 store 可以直接用 Composition API 风格写，和 Vue3 的风格统一。
>
> 我选 Pinia 是因为它是 Vue3 生态的官方推荐，API 更简单，代码量更少，适合这个项目的规模。

## 五、现场演示类

### ⭐ Q17：演示时可能被问：这个问题它为什么回答错了？

**参考回答**：

> 这个问题很好。大模型的回答质量受几个因素影响：
>
> 第一，知识库内容的质量——如果知识库里的描述本身不够清晰，模型也很难给出准确答案。
>
> 第二，提问方式——问题越具体，模型越容易找到对应内容。
>
> 第三，知识库长度——如果文档超过了我设置的 12000 字符上限，超出部分会被截断，相关内容可能丢失。
>
> 这也是我在展望里提到的改进方向——引入向量数据库做语义检索，可以更精准地找到相关片段，提升回答质量。

---

### Q18：系统的并发能力怎么样？能支持多少用户同时使用？

**参考回答**：

> 当前是开发模式，Flask 的内置服务器是单线程的，并发能力有限，主要用于开发和演示。
>
> 如果要生产部署，可以用 Gunicorn 多进程部署，配合 Nginx 反向代理，并发能力会大幅提升。
>
> 不过这个系统的瓶颈主要在 AI 接口的响应时间，每次问答需要等待大模型返回，这个时间取决于智谱 AI 的接口性能，一般在 3～10 秒，这是外部依赖，不是系统本身的问题。

---

### Q19：如果让你重新做这个项目，你会有哪些不同的选择？

**参考回答**：

> 主要有两点：
>
> 第一，知识库处理方面，我会引入向量数据库（比如 Chroma）和文本分块技术，把文档切成小段分别向量化存储，提问时只检索最相关的片段传给模型，这样可以处理更大的文档，回答也更精准。这是目前主流的 RAG（检索增强生成）方案。
>
> 第二，知识库格式方面，我会加入 PDF 解析支持，因为实际使用中 PDF 比 TXT 更常见。
>
> 当然，这些改进会增加系统复杂度，对于毕设的时间限制来说，当前的方案是合理的权衡。

## 六、送分题（一定要答好）

### ⭐ Q20：你在这个项目中学到了什么？遇到了哪些困难？

**参考回答**：

> 收获很多，主要有三点：
>
> 第一，**工程实践能力**：这是我第一次完整做一个前后端分离的项目，从需求分析、数据库设计、接口设计到前端实现，走了一遍完整的开发流程，对软件工程的理解更深了。
>
> 第二，**AI 接入经验**：学会了如何调用大模型 API，以及 Prompt 工程的基本方法，这在当前 AI 快速发展的背景下很有价值。
>
> 第三，**问题排查能力**：开发过程中遇到了不少问题，比如 Python 包版本冲突、macOS 端口占用、文件编码兼容等，通过查文档、看报错信息一步步解决，锻炼了调试能力。
>
> 最大的困难是 Prompt 工程的调试——怎么写 System Prompt 才能让模型既准确又不死板，这个需要反复测试和调整，没有固定答案，需要靠经验积累。
